EmpProject GROUP BY empNo;RESULT: this query is problematic.totalHours is not in the GROUP BY clause and is not wrapped in an aggregate like SUM(). When an employee has more than one row in the view (e.g. E2 has totalHours 8 and 5 for two different projects) the database doesn't know which value to return. Strict databases like PostgreSQL and SQL Server would throw an error. MariaDB in strict mode will also reject it. The correct way to get total hours per employee across all projects is:SELECT empNo, SUM(totalHours) AS totalHoursFROM   EmpProjectGROUP  BY empNo;RESULT:empNo  totalHours-----  ----------E1     6E2     13E3     7E4     13E5     3E6     6E7     8E8     57.28ANSWER:A materialized view saves the query result as a real table so you don't have to rerun the query every time.For ExpensiveParts, it stores part numbers where partCost is over 1000. Keeping it updated depends on what changes in Part:INSERT or price increase above 1000 just add the partNo to the view. No need to check the rest of Part.DELETE or price drop below 1000 you cannot just remove the partNo straight away. The same part might exist under another contract that is still over 1000, so you have to check Part first before removing it.This is because one part can appear in multiple rows under different contracts, and the view only keeps distinct part numbers.
